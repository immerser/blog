kafka官方文档阅读

介绍

  1. kafka的消息不管是否被消费都会维护指定的时间
  2. partition作用

每个分区都有一个leader其他均为follower leader负责读写 follower负责复制,leader失败后会再次选举

>突破单机存储限制

>并行处理

>备份
  1. producers

负责往topic中的partition写入数据,支持轮询写入partition内

  1. consumers
    消费者以组的形式进行消费数据,每一个topic内的数据会被一个特定的消费者组内的一个消费者消费,一个topic可以被多个消费者组消费

    同一个消费者组可以进行负载均衡
    多个消费者组可以进行数据广播处理
    仅支持topic内的消费顺序
    可以只设置一个分区进行整个topic的顺序

应用场景

  1. 消息系统
  2. 存储系统
  3. 流处理系统

api设计

  1. producer 发送数据
  2. consumer 消费数据
  3. streams 输入topic到输出topic
  4. connect 允许从源系统连续输出

配置

  1. broker configs
    1. 必须配置的属性
      broker.id log.dirs zookeeper.connetct
  2. producer configs
    1. 负载均衡
    2. 异步发送

  3. consumer configs

    1. 推拉模型
    2. 消费位置

  4. connetc configs

  6. stream configs

解释

  1. 消息投递系统
    1. 最多一次

和influxdb区别

Where InfluxDB is better:

If you’re doing event logging.
Commercial option offers clustering for InfluxDB, which is also better for long term data storage.
Eventually consistent view of data between replicas.
Where Prometheus is better:

If you’re primarily doing metrics.
More powerful query language, alerting, and notification functionality.
Higher availability and uptime for graphing and alerting.
